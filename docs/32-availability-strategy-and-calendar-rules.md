# 32 — Availability Strategy and Booking Calendar Rules

This document completes Phase 6.1 and defines the rules that the availability service and public booking calendar must follow.

## Phase

```text
Phase: Phase 6 — Availability Calendar Foundation
Subphase completed by this document: 6.1 Availability strategy and booking calendar rules
Next subphase: 6.2 Availability domain service foundation
```

## Goal

Phase 6.1 is not the final calendar feature. It creates the rule contract and typed code foundation that Phase 6.2 and Phase 6.3 must use.

This subphase adds:

```text
Typed availability date ranges
Typed availability block sources
Composed listing dependency rules
Date-only helpers for YYYY-MM-DD values
Preparation buffer range helpers
Documentation for public calendar boundaries
```

It does not add checkout, payments, email, iCal sync, database writes, migrations, or PMS functionality.

## Date Range Convention

TRP Booking uses the standard lodging convention:

```text
checkInDate: inclusive
checkOutDate: exclusive
```

Example:

```text
checkInDate = 2026-08-10
checkOutDate = 2026-08-12
```

The stay occupies the nights of:

```text
2026-08-10
2026-08-11
```

The guest leaves on:

```text
2026-08-12
```

## Availability Sources

Availability must eventually be evaluated from:

```text
CONFIRMED direct reservations
Active PENDING_PAYMENT reservations that have not expired
Manual blocks
Maintenance blocks
Imported Airbnb blocks
Composed listing dependency blocks
Preparation buffer blocks that have not been manually unlocked
```

Phase 6.1 does not query the database yet. Phase 6.2 must use these sources when it introduces the domain service.

## Composed Listing Rules

The project has two individual accommodations and one composed listing.

```text
Apartamento Blanco y Negro
Bungalow Refugio Perfecto
Refugio Completo
```

Availability dependency rules:

```text
Reserve or block Apartamento Blanco y Negro
-> Apartamento Blanco y Negro unavailable
-> Refugio Completo unavailable

Reserve or block Bungalow Refugio Perfecto
-> Bungalow Refugio Perfecto unavailable
-> Refugio Completo unavailable

Reserve or block Refugio Completo
-> Refugio Completo unavailable
-> Apartamento Blanco y Negro unavailable
-> Bungalow Refugio Perfecto unavailable
```

These rules are implemented in:

```text
lib/availability/rules.ts
```

## Preparation Buffer Rules

Preparation buffers must preserve the operational policies already defined for each accommodation:

```text
Apartamento Blanco y Negro: 1 day before and 1 day after
Bungalow Refugio Perfecto: 2 days before and 2 days after
Refugio Completo: 2 days before and 2 days after
```

Preparation buffer range convention:

```text
Before check-in buffer: [checkInDate - daysBefore, checkInDate)
After check-out buffer: [checkOutDate, checkOutDate + daysAfter)
```

Example for Apartamento Blanco y Negro with one day before and one day after:

```text
Stay: 2026-08-10 to 2026-08-12
Before buffer: 2026-08-09 to 2026-08-10
After buffer: 2026-08-12 to 2026-08-13
```

These ranges are generated by:

```text
buildPreparationBufferRanges()
```

## Public Calendar Boundary

Phase 6 must create a public availability foundation, not a checkout flow.

Allowed in Phase 6:

```text
Public date visibility
Disabled unavailable dates
Availability service foundation
Preparation buffer evaluation
Manual block evaluation foundation
Confirmed and active pending reservation evaluation foundation
```

Not allowed in Phase 6:

```text
Creating real reservations from public UI
Starting payment
Tilopay integration
Resend integration
Airbnb iCal import/export
Admin calendar management UI unless explicitly introduced later
PMS features
Guest date modification for confirmed reservations
```

## Implementation Notes for Phase 6.2

Phase 6.2 should build the availability domain service foundation using the rules added in Phase 6.1.

Expected direction:

```text
Read reservations and calendar blocks through Prisma.
Normalize date-only ranges.
Apply composed listing dependency rules.
Exclude expired pending reservations.
Exclude soft-deleted calendar blocks.
Ignore unlocked preparation buffer blocks.
Return a typed availability result for a requested accommodation and date range.
Do not create reservations yet.
Do not start payment yet.
```
