# AGENTS.md — TRP Booking Collaboration Rules

This file defines the working rules for AI-assisted development in the `trp-booking` repository.

## Project Identity

- Technical name: `trp-booking`.
- Internal name: `TRP Booking`.
- Commercial brand: `Tu Refugio Perfecto`.
- Public SEO name: `Bungalows Tu Refugio Perfecto`.
- Official domain target: `turefugioperfecto.com.gt`.

## Scope Rule

TRP Booking is a **direct booking website**, not a PMS.

Do not expand this project into modules that belong to TAMIAS, such as:

- Maintenance operations.
- Inventory management.
- Purchases.
- Internal documents.
- Full property operations.
- Internal AI assistant features.
- Full multi-organization PMS workflows.

TAMIAS remains the PMS / internal operations system. TRP Booking remains focused on public reservations, payments, Airbnb calendar synchronization, and the minimal admin area required for this flow.

## Documentation-First Rule

Before implementing new functionality:

1. Review the relevant document in `/docs`.
2. Check the current phase in `docs/10-phases.md`.
3. Confirm that the change fits the project scope.
4. Update documentation if the decision changes.
5. Then implement code.

## Repository Review Rule

When reviewing the GitHub repository, always verify the real latest state and avoid stale cached assumptions. Use cache-busting when fetching repository files or public GitHub content.

## Secrets and Sensitive Data

Never commit secrets or operational tokens to the repository.

This includes:

- Tilopay credentials.
- Resend API keys.
- Cloudinary API secret.
- Database URLs.
- Auth secrets.
- Cron secrets.
- Airbnb iCal URLs with tokens.
- Webhook secrets.

Airbnb iCal URLs must be stored privately through environment variables, admin configuration, or database records protected by admin authentication. They must not be placed in README files, public docs, frontend code, or committed seed files.

## Payment Safety Rules

- Never store card data.
- Never trust prices sent from the frontend.
- Always calculate totals on the server.
- Always validate availability on the server.
- A reservation is not confirmed just because the guest returns from the payment page.
- A reservation is confirmed only after a trusted payment confirmation, preferably through Tilopay webhook verification.
- Payment webhook endpoints must validate signatures/secrets.
- Refunds must be auditable.

## Airbnb iCal Rules

- Sync Airbnb iCal imports every 30 minutes using Vercel Cron in production.
- Provide an admin button to force synchronization manually.
- Use a protected endpoint for cron execution.
- Do not expose iCal import URLs publicly.
- Export iCal URLs generated by this system can be public but must use unguessable tokens.

## Code Quality Rules

Use:

- TypeScript strict mode.
- Zod or equivalent validation for inputs.
- Prisma migrations for database changes.
- Server-side validation for business rules.
- Feature-based organization.
- Reusable UI components.
- Centralized environment validation.
- Clear error handling.
- Tests for availability, overlapping reservations, pricing, cancellation, and refunds.

Avoid:

- Business logic inside UI components.
- Duplicated payment logic.
- Hardcoded secrets.
- Native browser `confirm()` for important admin actions.
- Unjustified dependencies.
- Unreviewed changes to payment or reservation flows.


## Centralized Error Message Rules

- All user-facing error messages must be centralized and reusable.
- Do not hardcode random error strings inside components, server actions, API routes, services, providers, or repositories.
- Use typed error keys grouped by domain, such as `reservation`, `payment`, `calendar`, `auth`, `upload`, and `validation`.
- Public error messages must support Spanish and English from the beginning.
- Never expose raw provider, database, payment, webhook, or stack trace errors to public users.
- Admin-facing errors may include more operational context, but must never expose secrets, tokens, credentials, full iCal URLs, or sensitive payloads.

## Reservation Change Rules

- Guests must not be able to freely modify confirmed reservation dates from the public website.
- Date changes require admin authorization, or the guest must cancel and create a new reservation according to the cancellation policy.
- Any admin-approved date change must re-check availability on the server before being accepted.
- Stay extensions while the guest is already on the property must check availability first.
- If an extension or date change increases the total, the additional amount must be paid or recorded before the updated dates are treated as confirmed.
- If a date change reduces the total, refund eligibility must follow the documented cancellation/refund policy and be auditable.
- All date changes, extensions, cancellations, and manual adjustments must create audit logs.

## Preparation Buffer Rules

- Confirmed reservations and imported Airbnb bookings must create automatic preparation buffer blocks.
- `Apartamento Blanco y Negro` requires 1 preparation day before and 1 preparation day after the stay.
- `Bungalow Refugio Perfecto` requires 2 preparation days before and 2 preparation days after the stay.
- `Refugio Completo` requires 2 preparation days before and 2 preparation days after the stay.
- Preparation buffers must block availability in TRP Booking and must be included in exported iCal availability for Airbnb.
- Admin users must be able to manually unlock preparation buffer days when operationally convenient.
- Manual unlocks of preparation buffers must be audited.

## Delivery Preferences

- Prefer real files in ZIPs respecting the project folder structure.
- Do not provide `.ps1` or `.sh` scripts as the primary way to apply changes unless explicitly requested.
- For small surgical changes, provide exact file paths and exact snippets to replace.
- Do not invent fields, entities, or requirements not present in documentation or confirmed by the user.

## Language and Tone

- Public site must support Spanish and English from the beginning.
- Admin copy can start in Spanish if needed, but should be prepared for i18n.
- The public tone should be warm, trustworthy, calm, natural, and professional.

## UI and Design System Rules

- Do not use unstyled native browser UI for user-facing booking, payment, calendar, modal, select, alert, confirmation, or form interactions.
- Do not use native `alert()`, `confirm()`, or `prompt()`.
- Do not expose native browser date pickers as the main booking calendar.
- All public-facing and admin-facing interactions must use the project design system.
- Use styled, accessible components based on the approved UI stack.

## Deletion Rules

- Do not hard-delete reservation, payment, refund, guest, calendar, or sync history data.
- Use soft delete for admin-managed business records unless a document explicitly allows hard delete.
